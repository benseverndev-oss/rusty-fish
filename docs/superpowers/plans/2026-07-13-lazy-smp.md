# Lazy SMP Multithreading Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add cooperative multi-threaded search (Lazy SMP) that scales with cores through a shared transposition table, leaving the single-threaded path unchanged.

**Architecture:** A sharded, lock-based `SharedTranspositionTable` wraps the existing `TranspositionTable`. `Searcher` holds it behind an `Arc`. With `Threads > 1`, the primary thread runs the current iterative-deepening loop while helper threads deepen independently over the shared table and stop flag.

**Tech Stack:** Rust 2024, existing Rusty Fish alpha-beta/PVS search, `std::thread`/`std::sync`, GitHub Actions.

## Global Constraints

- `Threads <= 1` must reproduce the current behaviour exactly.
- Only the primary thread bumps the shared generation, probes Syzygy/book, and reports `info`.
- No nested shard locks and no lock held across recursion.
- Workspace tests, tactical suite, fixed-opponent gauntlet, and CodeQL are required remote evidence; a `Threads`-varied external SPRT is the strength gate.

```rust
fn default_search_options_use_one_thread() {
    assert_eq!(SearchOptions::default().threads, 1);

#[test]
fn shared_transposition_table_round_trips_across_shards() {
    let table = SharedTranspositionTable::new(4_096);
    for key in [1u64, 1 << 48, (7 << 48) | 9] {
        table.store(key, TranspositionEntry {
            depth: 4, score: 12, bound: Bound::Exact, best_move: None,
        });
        assert_eq!(table.get(key).map(|entry| entry.score), Some(12));
    }
}
```
