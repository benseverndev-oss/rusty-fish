# Phase 3 (Uncertainty) & Phase 4 (Policy) — Scope

Status: scoping only, not approved to build. Written 2026-07-24 after Phase 2
shipped (`docs/RESEARCH-DIRECTION.md`). Each phase below is its own spec → plan →
build → gate cycle, roughly Phase-2-sized. Do not start either without its own
design pass.

## What Phase 2 established (and what it cost)

Shipped: learned LMR, **+38.3 Elo** (equal movetime, 4096 games, AcceptH1), adopted
as the engine default. The reusable machinery: telemetry → dataset → tiny model →
hot-loop integration (byte-identical guarded) → sharded equal-movetime SPRT gate.

Three measured findings that should shape everything after:

1. **The models saturate on data, fast.** Learned LMR hit val AUC ~0.94 and stayed
   there across depth-8 vs depth-9 data (0.9390 / 0.9378) and hidden-16 vs hidden-64
   (0.9378 / 0.9396). Not data-limited, not capacity-limited — **feature-limited**.
   (The NNUE showed the same shape one level down: 3M→17M positions gave ~0 Elo.)
2. **Classical heuristics are better-placed than they look.** Expanding the model's
   regime to *introduce* reductions gated +36.5 vs the adjust-only +38.3 — no gain.
   The classical `move_index >= 3` eligibility is already well-tuned. The learned
   win came from *how deep to reduce*, not *where to reduce*.
3. **Equal-movetime is the honest gate for search changes.** Equal-nodes both
   overshoots badly (the budget is only checked between depth iterations, ~20x) and
   under-credits a change that makes the search cheaper per node.

## Cross-cutting prerequisite: richer telemetry features

Finding (1) says the single highest-leverage next work is **not** another phase — it
is widening the telemetry feature set, which raises the ceiling for *every* learned
component. Current features are 10 thin context scalars (depth, ply, move_index,
is_quiet, is_priority, pv_node, gives_check, static_eval, extension, reduction).

Candidates to add (all cheap to capture at the existing hook): history / counter-move
/ killer scores for the move, SEE value, move flags (capture, promotion, castle),
moved piece kind, victim kind, TT-move/TT-depth/TT-bound context, phase, and
material balance. This is a Phase-1 schema extension plus a retrain, and it is the
prerequisite that makes Phases 3 and 4 worth doing well.

**Recommendation: do this before either phase.**

## Phase 4 (Policy / move ordering) — cheaper, do it first

Despite the numbering, Phase 4 is the *cheaper* next phase, because **its label
already exists**: the telemetry records `caused_cutoff` per move. A cutoff predictor
can be trained on data we already have.

- **Target:** P(move causes a beta cutoff) — or P(move is best) at the node.
- **Use:** augment move ordering (root, quiet, LMR ordering, singular candidates).
  Ordering compounds: better first moves mean more cutoffs mean smaller trees.
- **The hard constraint:** ordering scores *every move at every node* — far hotter
  than the LMR correction, which only fires on eligible moves. A per-move MLP may be
  too slow to pay for itself. Design must start from an inference budget, not from
  model quality. Options: a very small net, quantised int8, a linear model, or
  scoring only the top-K candidates from the existing heuristic ordering.
- **First step (offline, no hot-loop risk):** train the cutoff predictor on existing
  telemetry and measure *ordering quality offline* (e.g. mean index of the actual
  cutoff move under learned vs current ordering). Only touch the search if the
  offline gain is large enough to plausibly survive the inference cost.
- **Gate:** equal-movetime sharded SPRT, same harness. Node count is the tell — good
  ordering should shrink trees at equal strength.

## Phase 3 (Uncertainty-aware search) — needs new data

- **Target:** how much the evaluation moves under deeper search (does the score or
  the best move change from depth d to depth d+k?), i.e. a per-position volatility
  estimate.
- **Requires a NEW dataset**: search each position at depth d and d+k and record the
  delta. That is a fresh generation pass (reuse the `gen_telemetry` fan-out shape),
  and it is more expensive than Phase 4's, which needs no new data.
- **Output:** either a second tiny model over search context, or an extra head on the
  NNUE. Prefer a separate small model first — it keeps the NNUE's byte-identical
  contract intact and is independently gateable.
- **Uses (in increasing risk):** aspiration-window sizing (safest, cheap, easily
  reverted) → time allocation at the root → extension/reduction modulation →
  quiescence depth.
- **First step:** aspiration windows only. It is the most isolated consumer, and a
  bad model costs re-searches rather than tactical blunders.
- **Gate:** equal-movetime sharded SPRT.

## Non-negotiables carried forward

- **Byte-identical invariant.** Every learned component ships behind a toggle whose
  off-state is provably identical to the classical path (best move, score, *and* node
  count), test-enforced. This is what let Phase 2 land safely.
- **Clamp the corrections.** Classical search stays the framework; learned outputs
  adjust it within bounds.
- **Observational data must observe the baseline.** `gen-search-telemetry` disables
  the learned model so datasets describe the behaviour being corrected, not the
  corrector's own output — otherwise retraining chases its own tail.
- **Gate before adopting, on equal movetime, with enough games.** 4096 games at
  50 ms/move via the sharded gate resolves a ~±5 Elo question; a few hundred does not.

## Suggested order

1. Richer telemetry features (+ retrain learned LMR on them) — raises every ceiling.
2. Phase 4 policy, offline-first, then hot-loop only if the inference budget allows.
3. Phase 3 uncertainty, starting with aspiration windows.
